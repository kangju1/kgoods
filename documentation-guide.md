# 문서 구조·관리 가이드

백엔드(Django API)·프론트(Next.js) 구축·고도화·유지보수를 위해 **무엇을 어디에 두고, 언제 고칠지**에 대한 팀 기준이다.

---

## 1. 원칙

| 원칙 | 설명 |
|------|------|
| **단일 진실 원천** | 동일 내용을 두 곳에 복붙하지 않는다. 한 곳에 두고 나머지는 링크한다. |
| **행동 변화 = 문서 PR** | API 경로·인증·데이터 모델·배포 방식이 바뀌면 **같은 PR 또는 직후 PR**로 문서를 갱신한다. |
| **독자별 계층** | 제품/도메인(`business`, `overview`, `strategy`)과 구현(`technical`, 코드 옆 README)을 섞지 않는다. |
| **시간이 지난 문서 표시** | 큰 결정을 바꿨으면 상단에 `> 이전 결정: … (날짜)` 한 줄 또는 ADR 링크를 남긴다. |

---

## 2. 현재 `docs/` 역할 (유지)

| 경로 | 용도 |
|------|------|
| `overview/` | 왜 만드는지, 벤치마크, 이해관계자 |
| `business/` | 업무 프로세스, 상품·거래처 도메인 언어 |
| `strategy/` | 로드맵, 리스크·고려사항 |
| `technical/` | 아키텍처, 스택, 데이터 모델, 자동화 방향 |

**구현 세부**가 길어지면 `technical/`에만 넣기 어려울 때 아래 **확장**을 고려한다.

---

## 3. 저장소가 늘어날 때 권장 구조

**모노레포**(예: `apps/api`, `apps/web`, `docs`)를 쓰는 경우:

- **`docs/`** — 제품·아키텍처·크로스컷팅(지금과 동일 계열)
- **`apps/api/README.md`** (또는 `backend/README.md`) — 로컬 실행, 마이그레이션, 테스트 명령
- **`apps/web/README.md`** — Node 버전, 환경 변수, 빌드/린트
- **API 계약** — Django에서 생성한 **OpenAPI(JSON/YAML)** 를 리포지토리에 커밋하거나 CI 아티팩트로 보관하고, 프론트는 이를 기준으로 타입·클라이언트 생성(가능 시)

**레포가 분리**(api / web / docs)되는 경우:

- `docs` 레포는 지금의 `overview`·`business`·`strategy`·고수준 `technical`에 집중
- 각 코드 레포 README에 **실행·배포·환경 변수**를 둔다
- API 계약은 **API 레포** 또는 **게이트웨이**가 소유하고, 웹 레포는 서브모듈·패키지·CI 복사로 동기화

---

## 4. API 문서 (특히 중요)

- **정본**: 가능하면 **스키마에서 자동 생성**(예: DRF + `drf-spectacular` → OpenAPI).
- **수동 문서**(`mvp-stack.md`의 엔드포인트 표 등)는 **요약·흐름**용으로 두고, 필드·에러 코드는 OpenAPI와 **충돌 시 OpenAPI 우선**.
- 버전: URL 버전(`/api/v1/`)과 문서 버전을 맞춘다.

---

## 5. 아키텍처 결정 기록 (ADR, 권장)

반복해서 논의되는 결정(JWT 보관 방식, 장바구니 정책, 배포 토폴로지 등)은 짧게라도 남긴다.

- 경로 예: `docs/adr/NNNN-title.md` (또는 `technical/decisions/`)
- 한 파일에 **맥락 → 결정 → 결과(트레이드오프)** 만 적는다. 장문 불필요.
- `technical/architecture.md` / `mvp-stack.md`에서는 ADR로 링크만 건다.

---

## 6. 운영·장애 (선택, 성장 시)

- `docs/runbooks/` — 배포 절차, 롤백, DB 백업, 자주 나는 장애 대응
- 비밀 값·계정은 문서에 쓰지 않고 **이름만** (실제 값은 비밀 저장소)

---

## 7. PR·리뷰 습관

- [ ] 공개 API·인증·스키마 변경 → OpenAPI 또는 동등 산출물 갱신
- [ ] 마이그레이션·필드 추가 → `data-model.md` 또는 ER 다이어그램/설명
- [ ] 새 환경 변수 → 해당 앱 README + (필요 시) 배포 문서
- [ ] “왜 이렇게 했는지”가 1년 후 필요하면 → ADR 한 장

---

## 8. Docsify(현재)와의 관계

- 마크다운은 **Git이 버전 관리**이므로, Docsify는 **읽기 UI**로만 본다.
- 장문 스펙은 PDF보다 **리포지토리 링크 + 검색 가능한 MD**가 유지보수에 유리하다.

---

## 9. 정리

| 목적 | 어디에 |
|------|--------|
| 비즈니스·프로세스 | `business/`, `overview/`, `strategy/` |
| 시스템 방향·스택 | `technical/architecture.md`, `technical/mvp-stack.md` |
| 엔티티·용어 | `technical/data-model.md` |
| API 필드·에러 | OpenAPI(자동 생성) |
| “왜 그렇게 했나” | `docs/adr/` |
| 로컬 실행·배포 커맨드 | 각 앱 `README.md` |
| 장애·배포 절차 | `docs/runbooks/` (필요 시) |

이 파일 자체는 **운영 규칙이 바뀔 때**만 수정한다.

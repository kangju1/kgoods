# 시스템 아키텍처

## 0. 확정된 기술 결정 (MVP 기준)

| 항목 | 결정 |
|------|------|
| 백엔드 | **Django**, **순수 RESTful API** (템플릿·서버 렌더링 없음) |
| 프론트엔드 | **Next.js** — **바이어 포털**과 **운영 어드민 UI** 모두 Next에서 구축 (별도 앱/라우트 그룹) |
| 도메인 분리 | 기능별 **Django app** 분리, 계정은 **`AbstractUser` 상속 `User`** |
| 인증 | **JWT** (액세스·리프레시; 구체는 [MVP 스택·API 설계](./mvp-stack.md)) |
| 상품 입력 | **이메일 파싱 미구축**. 관리자가 **어드민(Next) + API**로 카탈로그 **직접 등록** |

이후 절은 위 전제에 맞춰 정리한다. 이메일 파싱·외부 자동 수집은 [자동화 전략](./automation.md)상 **후속 페이즈**로 둔다.

## 1. 플랫폼 구성 개요

```
┌──────────────────────────────────────────────────────────────┐
│                     Next.js (단일 리포지토리 권장)              │
│  (public) 바이어 포털          │  (protected) 운영 어드민        │
│  카탈로그·상세·가입·주문        │  상품·마스터·주문·거래처 관리    │
└────────────────────────────┬─────────────────────────────────┘
                             │ HTTPS / JSON
                             ▼
┌──────────────────────────────────────────────────────────────┐
│                  Django REST API                              │
│  accounts │ catalog │ partners │ orders │ (추후 notifications) │
└────────────────────────────┬─────────────────────────────────┘
                             │
                             ▼
┌──────────────────────────────────────────────────────────────┐
│  PostgreSQL (권장)  │  객체 스토리지(상품 이미지, S3 호환)        │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│  후속 연동 (MVP 범위 밖): ERP·PG·알림톡·Nippan·이메일 파싱 등    │
└──────────────────────────────────────────────────────────────┘
```

## 2. 주요 모듈 (MVP 정렬)

### 2-1. 상품 카탈로그 (`catalog` app)

- 관리자가 **수동** 등록·수정·삭제 (draft → published 워크플로 권장)
- 카테고리 트리, 아티스트 마스터, 이미지(다매체), **SEO용 slug·메타**
- **공개 목록/상세**는 `published` + 정책에 따른 가시성(성인·비공개 등)
- 재고는 MVP에서 **수동 수치** 위주 (`Product`/`Variant` 수준)

### 2-2. 거래처·가격 (`partners` app)

- 기존 B2B **Partner**, **가격 티어**, **상품×티어별 공급가**
- **User ↔ Partner** 선택적 연결: 잠재 바이어는 Partner 없이 시작, 기존 거래처 계정은 연결 가능

### 2-3. 주문 (`orders` app)

- **장바구니**(비회원 세션 + 로그인 후 병합 권장 — [mvp-stack.md](./mvp-stack.md))
- 주문 접수 → 어드민에서 조회·상태 변경(견적·발주는 수기 병행)
- **MVP 결제**: PG 미연동 시 `payment_status`는 **견적/후불 전제** 값으로 단순화

### 2-4. 계정 (`accounts` app)

- `AbstractUser` 기반 **이메일 중심** 가입, **이메일 인증** 후 주문 가능(권장)
- 선택 필드: 전화번호, 회사명 등(바이어 프로필)
- **스태프**(`is_staff`)만 어드민 API/화면 접근

### 2-5. 알림 (후속)

- 주문 접수 메일 등은 MVP에서 **트랜잭션 메일 최소** 또는 수동 연락으로 시작 가능

### 2-6. 신보 이메일 파싱

- **현 단계 비구현**. 동일 목표는 **관리자 수동 입력**으로 대체. 자동화는 Phase 2+ ([automation.md](./automation.md)).

## 4. 기술 스택 요약

| 영역 | 제안 |
|------|------|
| API | Django REST framework, OpenAPI 스키마 생성 권장 (`drf-spectacular` 등) |
| 인증 | JWT, 권한 분리: 공개 읽기 / 로그인 쓰기 / 스태프 전용 admin API |
| 비동기 | MVP는 동기 위주; 이메일 발송·대량 작업 도입 시 Celery 검토 |
| Next.js | App Router, 바이어·어드민 **라우트 그룹** 분리, 상품 상세는 **ISR/SSG + revalidate** 권장 |
| 저장소 | PostgreSQL, 미디어는 S3 호환 + CDN |

세부(토큰 보관, 엔드포인트 그룹, Next 라우트)는 [mvp-stack.md](./mvp-stack.md) 참고.

## 5. 데이터 흐름 (MVP — 수동 카탈로그)

```
[운영자: Next 어드민]
    │
    ▼ REST API
[Django: Product draft → published]
    │
    ├─► [Next: 공개 상품 페이지 SEO 노출]
    │
    └─► [바이어: 가입 → 장바구니 → 주문 접수]
            │
            ▼
    [Django: Order RECEIVED]
            │
            ▼
    [운영자: 어드민에서 확인 → 수기 견적·발주]
```
